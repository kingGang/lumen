## 1. 收敛后的单一真相（经对抗审查，四份规格统一到这里）

### 协议（`lumen-protocol/src/remote.rs`）

```rust
// RemoteFrame 新增两变体（仿现有 ListDir/ListDirResult 风格，字段名对齐：用 path 不用 root_path）
ListDirRecursive {
    req_id: u64,
    path: String,        // 被控端不透明根目录路径
    show_hidden: bool,
    max_depth: u32,      // 0 = 无穷；根目录深度=1
    max_entries: u32,    // 0 = 无穷
},
ListDirRecursiveResult {
    req_id: u64,
    path: String,        // 原样回带
    entries: Vec<RecursiveDirEntry>,  // DFS 平铺序（见不变量）
    truncated: bool,
    err: Option<FsErr>,
},

// 新类型
pub struct RecursiveDirEntry {
    pub rel_path: String,  // 相对 root，【正斜杠 '/' 分隔】（跨平台统一；COM 侧转 '\'）
    pub path: String,      // 被控端不透明绝对路径（FetchReq 用；任何一端都不拆）
    pub is_dir: bool,
    pub size: u64,         // 目录为 0；文件填实际大小（descriptor FD_FILESIZE 必需，否则 0KB bug）
}

// 常量（与 4 MiB 单帧上限自洽：10000 项 × ~140B JSON ≈ 2 MB < 4 MiB）
pub const LIST_DIR_RECURSIVE_MAX_ENTRIES: u32 = 10_000;
pub const LIST_DIR_RECURSIVE_MAX_DEPTH: u32 = 20;
```

- **路径不加 base64**：rel_path/path 是文本，JSON 原生支持，仅 ~5-10% 转义膨胀；base64 反而更糟。
- **不变量（写进 `RecursiveDirEntry` 文档注释）**：`entries 数组顺序 == descriptor fgd 顺序 ==
  lindex 索引`，全链路 **DFS**、**严禁任何一层重排**。

## 2. 实现时必守的风险 checklist（来自对抗审查）

| 编号 | 风险 | 必守做法 |
|---|---|---|
| **H3/H4** | lindex 错位 / 空目录建不出 | **DFS** 遍历，父目录项必在其子项之前；空目录单独成 descriptor 项；全链路不重排 |
| **H5** | 复制后立即 Ctrl+V 粘到旧文件 | `start_clip_dir` **第一行先 `svc.clear()`** 再发帧，再 toast「正在枚举…」 |
| **M1** | 枚举中切复制别的，旧应答污染 | 任何复制动作（单文件/本地/目录）都 `inflight_clip_dir = None` |
| **M2** | 大目录瞬间起几百条流 | `make_stream` 改**惰性**（首个 `Read` 才 `start_clip_fetch`，非 `GetData`）+ **全局 Clipboard fetch 并发闸**（4~8 条） |
| **M5** | size 丢失 → 0KB | `Notice::ClipDirReady(Vec<RecursiveDirEntry>)`（含 size）；`set_remote_dir` 收 `Vec<RecursiveDirEntry>`，内部转私有 `DescItem` |
| **M3/M4** | 截断/失败用户无感、剪贴板状态不明 | 截断/失败 toast 醒目 + **i18n 三语**（en/zh_cn/zh_tw）；失败路径确保剪贴板已 clear |
| **L1** | cItems=0 空 descriptor | cItems=0（空树/全截断）时**不 set 剪贴板**，退回 clear |
| **L2** | 深层 rel_path 超 260 | UTF-16 超 259 的项**整项剔除 + 计 truncated**，不中间截断成乱路径 |
| **L3** | 符号链接环 / canonicalize 慢 | `follow_links(false)`，符号链接目录当普通项不下钻，免 canonicalize+visited |
| **L4** | 超时中断 | 被控端 DFS 每弹一项查时钟 + 代次 epoch 双中断源 |
| **M6** | 隐藏项递归与浏览树不一致 | `show_hidden` 复用单层 ListDir 同一取值路径 |

## 7. 联调修复（2026-06-22）

**海风哥反馈**：「只把目录里面的文件复制过来了，没有把文件夹一起复制过来」。

**根因**：`rel_path` 是相对被复制目录**内部**的路径（不含目录名本身），descriptor 里是 `a.txt` /
`sub\b.txt`，资源管理器把内容直接铺到目标根、丢了顶层文件夹。

**修复**（`remote_ws.rs::apply_list_dir_recursive`）：控制端把被复制目录名（`selected_item` 的 name，
存进 `inflight_clip_dir: Option<(u64, String)>`）给整棵子树套「根目录名」顶层前缀，并在最前插入根目录
项 → descriptor 变成 `myfolder`、`myfolder\a.txt`、`myfolder\sub`、`myfolder\sub\b.txt`，粘贴出
「目标\myfolder\…」。顺带修了空目录（现在能粘出空的顶层文件夹）。`lindex` 仍对齐（顶层 / 子目录项不
被请求 `FileContents`，文件项 `abs_path` 不变）。新增 4 回归单测（前缀+顶层项 / 空目录 / 陈旧去重 /
失败），三连绿（clippy 0 warning、test 441+13、build 0）。待海风哥按 §6 清单重测。

## 8. 多文件 descriptor 联调（2026-06-22，"复制不了任何文件"根因 + 标准做法）

**现象**：复制非空目录粘贴到资源管理器，连文件都拉不到（单文件 OK）。诊断日志（`[片8]`）确认全链路
正常（被控端枚举 N 项、控制端构造虚拟文件目录），问题在 COM / 资源管理器侧。

**真因**（agent 查证 MSDN + WinSCP/7-Zip + Raymond Chen）：descriptor 里目录项的 `dwFlags` 没设
**`FD_ATTRIBUTES`(0x04)** → 资源管理器忽略 `dwFileAttributes` 里的 `FILE_ATTRIBUTE_DIRECTORY` →
把目录项当**文件**去 `GetData(FILECONTENTS)` 拉内容 → 拉空 → 整批失败。

**权威要点**：
- `CFSTR_FILECONTENTS` 的 `FORMATETC.lindex` 是 **`fgd[]` 数组绝对下标**（含目录项），不是文件序号。
  本实现一直用 `items.get(lindex)`（绝对下标），正确。
- 目录项带 `FD_ATTRIBUTES | FILE_ATTRIBUTE_DIRECTORY` → 资源管理器 `CreateDirectory`、**不** GetData
  该项（但它仍占一个 lindex 槽）。
- `QueryGetData` 须对任意合法 lindex 回 S_OK（本实现 `supports` 不看 lindex，符合）。
- `EnumFormatEtc` 里 FILECONTENTS 只枚举一份（lindex=-1）（本实现符合）。
- 文件项须带 `FD_FILESIZE` + 真实大小（否则 0KB / 截断）。

**修复路径**：
1. 先临时改「只列文件项 + 相对路径」绕过目录项（**海风哥验证非空目录通过**）——规避 lindex/FD_ATTRIBUTES，
   但丢空目录。
2. 再改回**标准做法**：列目录项 + 文件项，目录项补 `FD_ATTRIBUTES`、文件项加 `FD_ATTRIBUTES`，
   DFS pre-order（父目录在子项前）。支持空目录、与 WinSCP/7-Zip 对齐。

**后续可选**（agent 建议，非阻塞）：实现 `IDataObjectAsyncCapability` 让资源管理器在后台线程逐文件拉，
避免大目录粘贴时阻塞 Explorer 主线程（当前惰性流 + Read 在资源管理器线程阻塞，单文件/小目录无碍）。

来源：MSDN datascenarios / dataobject / FILEDESCRIPTORW；Raymond Chen「Dragging a virtual file」；
WinSCP dragext；7-Zip FileManager。
