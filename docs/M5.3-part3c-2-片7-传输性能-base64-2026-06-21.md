# M5.3 part3c-2 片7 传输性能优化（base64 编码）2026-06-21

> 承接「片6 虚拟文件剪贴板」。海风哥实测虚拟文件下载**验证通过**，但提出性能问题：
> 局域网两机，同一个 59 MB 文件，**Windows 远程桌面（RDP）复制比 lumen 快 ~3 倍**。
> 本片定位根因并修复，server 仍**零改**。

## 1. 问题与根因

- **现象**：lumen 下载 / 上传大文件明显慢于 RDP（海风哥实测 59 MB 约慢 3 倍）。
- **根因**：文件块字节 `FileChunk.data` / `PutChunk.data` 是 `Vec<u8>`，经 `serde_json` 序列化会变成
  **JSON 数字数组**（`[12,34,255,...]`）。每个字节平均要 `"123,"` ≈ 3.7 个字符表示 → **线缆传输量膨胀约 3.7 倍**。
  这与海风哥观察的「慢约 3 倍」高度吻合——瓶颈不在磁盘 / 协议往返，而在 **JSON 把二进制撑大了近 4 倍**。

## 2. 方案（三处优化，server 零改）

| 优化 | 旧 | 新 | 效果 |
|------|----|----|------|
| **块字节编码** | `Vec<u8>` → JSON 数字数组（~3.7x 膨胀） | `#[serde(with = "b64")]` base64 字符串（~1.33x 膨胀） | **传输量降到约 1/2.8**，接近原始二进制 |
| **单块大小 `FILE_CHUNK`** | 64 KiB | 256 KiB | 等量数据**帧数 / ACK 往返 / JSON 序列化次数降到 1/4** |
| **在途窗口 `FETCH_WINDOW`** | 4 | 8（= 2 MiB 在途） | 填满局域网 / 中等延迟链路的带宽时延积 |

- base64 单块：256 KiB → ~341 KiB 文本，**远小于 server 的 4 MiB 单帧 / 单消息上限**（`ws.rs` 的
  `MAX_WS_MESSAGE` / `MAX_WS_FRAME`），无需调 server。
- **server 为什么不用动**：`lumen-server` 对 Relay 载荷只做 `serde_json::Value` **盲转**，从不解析
  `FileChunk` 内部。base64 只改变 `lumen-protocol` 里 `Vec<u8>` ↔ JSON 字符串的表示，对 server 完全透明。
  **海风哥正在运行的 server 二进制无需重编 / 重启**，继续盲转新帧即可。

## 3. 改动清单

- `Cargo.toml`（workspace）：新增 `base64 = "0.22"`。
- `crates/lumen-protocol/Cargo.toml`：引用 `base64 = { workspace = true }`。
- `crates/lumen-protocol/src/remote.rs`：
  - `FileChunk.data` / `PutChunk.data` 加 `#[serde(with = "b64")]`。
  - `FILE_CHUNK` 64 KiB → 256 KiB；`FETCH_WINDOW` 4 → 8。
  - 新增私有 `mod b64`：`serialize`（`Vec<u8>` → base64 串）/ `deserialize`（base64 串 → `Vec<u8>`，非法
    base64 回 serde 自定义错误）。app / server 对此无感知（字段类型仍是 `Vec<u8>`）。
- `Cargo.lock`：base64 入锁。
- **未触碰**：server、app 业务代码（`remote_ws.rs` 的 `fetch_src_worker` 等用 `FILE_CHUNK` 常量分配
  缓冲，常量改了自动生效，无需改逻辑）。

## 4. 验证（自测）

- **clippy**：`cargo clippy --workspace --all-targets` → EXIT 0，**0 warning**。
- **test**：`cargo test` → 全绿（lumen-app 432 passed、lumen-protocol 等全部 0 failed）。base64 + 常量
  调优**未破坏任何既有测试**（含协议 round-trip）。
- **build**：`cargo build -p lumen-app` → EXIT 0；已 `Copy-Item` 刷新 `target\debug\lumen-debug.exe`。
  - 注：全 workspace build 会因 `lumen-server` 重链撞上「运行中 server 二进制被锁」报 LNK1318，但
    **server 本就不需要更新**，故只构建 `lumen-app` 即可，属正常。
