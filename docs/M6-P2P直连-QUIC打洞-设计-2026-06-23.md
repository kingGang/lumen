## 0. 一句话目标

控制端↔被控端 **直连**，绕开服务器中继 → **降延迟**（省一跳，接续刚修的输入延迟的「终极版」）+ **降服务器
带宽/成本**（文件传输尤其受益）。**中继这套不废**——是 P2P 打洞失败时的兜底，也是 P2P 的信令通道。

设计原话（`架构设计.md:125`）：**中继优先 + P2P 终局；P2P（QUIC 打洞 + 中继回退）作为 M5 之后的优化项，
M5 全中继跑通、形态稳定后再做。** 现在形态已稳定（M5.3 全系列两机验收通过），可以开做。

## 1. 为什么现在能干净地做（集成面有多小）

接手前先理解：**P2P 不是重写传输，是在现有数据面上叠一条更快的旁路**。集成点出奇地小，三条已勘定：

### 1.1 数据面只有一个收发口
- **发**：所有数据面帧（镜像输出/输入/文件/多窗格/回看……）统一过
  [`RemoteWs::send_frame`](crates/lumen-app/src/remote_ws.rs:3786) → `RemoteC2S::Relay(value)` → cmd_tx → WS。
- **收**：入站统一 `WsEvent::Server(RemoteS2C::Relay(value))` →
  [`apply_relay(&value)`](crates/lumen-app/src/remote_ws.rs:3802) → `RemoteFrame::from_value` → 状态机。

**P2P 集成 = 在 `send_frame` 加一次选路（P2P 通 → 走 QUIC，否则走 Relay）+ 把 QUIC 收到的帧喂回
`apply_relay` 的同一入口。** 镜像渲染、输入转发、文件传输、多窗格、回看——**上层全部零改动**。
这就是设计原话「数据面帧复用现有 `RemoteFrame`（传输层换掉即可）」的落地点。

### 1.2 TLS 栈已对齐，QUIC 零额外加密栈
- `Cargo.lock` 现有 **rustls 0.23.40 + ring 0.17**（tungstenite 的 `rustls-tls-native-roots` 带来）。
- **quinn 0.11** 正好对接 rustls 0.23 + ring 后端 → 不新增加密栈、版本天然对齐。
- QUIC 自带 **TLS 1.3** → 直连数据面**端到端加密天然就绪**（不经服务器，安全模型变化见 §6）。

### 1.3 信令复用现有 Relay 通路 → 服务器零改动
- 打洞需要的「交换公网端点候选」走**新增的 `RemoteFrame::P2pSignal`**，包进现有 `Relay` 盲转。
- **服务器照旧盲转 `serde_json::Value`，零改动**（前向兼容铁律，有单测守护）。
- 信令通道本身已通过 JWT + 配对鉴权，可作直连证书指纹的**信任锚**（见 §6）。

## 2. 技术选型

| 模块 | 选型 | 理由 |
|---|---|---|
| QUIC 栈 | **quinn 0.11** | rustls 0.23/ring 后端与现有对齐；成熟、纯 Rust；多路复用解队头阻塞 |
| 异步运行时 | **tokio current-thread**，隔离在 P2P 后台线程内 | quinn 需 tokio；隔离后主线程仍是同步 tungstenite 范式，**主线程模型零污染** |
| 端点发现(STUN) | **服务器兼任 STUN 反射**（自建 UDP echo 回显 observed addr），公共 STUN 作备选 | 自主可控、国内可达、不依赖第三方；与现有 server 同生命周期 |
| 证书 | **自签**（rcgen 生成临时证书），指纹经信令通道交换校验 | 无需 CA；信任锚是已鉴权的中继会话（§6） |

> **架构级决策（须海风哥拍板）**：引入 `quinn` + `tokio` 到 `lumen-app`（当前客户端刻意不依赖 tokio）。
> 代价：编译时间/二进制体积增加。收益：延迟/带宽终极优化。已在 §1.2 确认 TLS 栈不重复、tokio 已在
> workspace lock（server 用）。tokio 隔离方案见 §5，确保不侵蚀现有同步线程范式。

## 3. tokio/quinn 隔离方案（不污染主线程）

现有 `remote_ws.rs` 范式：**一条后台线程跑同步 tungstenite，channel 与主线程通信**（`cmd_tx`/`evt_rx`）。
P2P **完全沿用此范式**，新开一条 P2P 后台线程：

```
主线程(同步, RemoteWs)
   │  send_frame 选路：p2p_ready? → p2p_out_tx : cmd_tx(Relay)
   │  ▲ 入站统一回 evt_tx(WsEvent::Server(Relay)) → apply_relay
   ▼  │
P2P 后台线程
   └─ 线程内起 current-thread tokio Runtime，block_on 驱动 quinn：
        · 出站：p2p_out_rx.recv() → QUIC stream/datagram 写
        · 入站：QUIC 读 → evt_tx(WsEvent::Server(RemoteS2C::Relay(value)))  ← 复用现有入站路径！
        · 打洞状态机：STUN 探测 / 候选交换 / 同时打洞 / 握手 / 回退
```

**关键优雅点**：
- 入站 P2P 帧**复用现有 `WsEvent::Server(RemoteS2C::Relay(v))`** → 经 `apply_server` → `apply_relay`，
  **零新增分发逻辑**，与中继帧自然合流（同一状态机，不区分来路）。
- 出站选路靠 `p2p_ready: Arc<AtomicBool>`（P2P 线程握手成功置位，主线程 `send_frame` 读），
  `send_frame` 仍可 `&self`，**签名不变**。
- tokio 被关进 P2P 线程的 `Runtime`，主线程与既有 WS 线程**完全不感知 tokio**。

## 4. 信令协议 + 打洞状态机

### 4.1 信令帧（Phase 0 已落地）
新增 `RemoteFrame::P2pSignal { kind, payload }`，走 Relay 盲转。`kind` 区分阶段，`payload` 为阶段数据：

- `Offer`：发起方公布候选端点列表 + 自签证书指纹 + 随机 nonce。
- `Answer`：应答方公布自己的候选 + 指纹。
- `Ready`：某条候选 QUIC 握手成功，确认选定直连。
- `Fallback`：打洞超时/失败，宣告继续全中继（对端停止打洞尝试）。

候选端点 = `[本地 LAN 地址(:port), STUN 反射得到的公网映射地址(:port)]`（多候选，逐个尝试）。

### 4.2 打洞流程（会话建立后触发）
1. **触发**：`SessionStarted` 后，双方各自绑定一个 UDP socket，向 STUN 反射端点查自己的公网映射。
2. **收集候选**：本地枚举网卡 LAN 地址 + STUN 回显公网地址 → 候选列表。
3. **交换**：控制端发 `Offer`（候选+指纹+nonce），被控端回 `Answer`（候选+指纹）。
4. **同时打洞**：双方**同时**向对端每个候选发 QUIC initial（同时发包穿透 NAT 映射；谁先到达对端
   NAT 已开洞）。
5. **握手 + 指纹校验**：任一候选 QUIC(TLS1.3) 握手成功 **且** 对端证书指纹 == 信令交换的指纹 → 选定。
6. **切换**：发 `Ready`，置 `p2p_ready=true`，数据面平滑切到 QUIC。
7. **回退**：超时（如 5s）无候选通过 → 发 `Fallback`，`p2p_ready` 保持 false，继续全中继。

### 4.3 失败与回退（中继永不断）
- **不变量**：中继 WS 连接**全程保持在线**——既是信令通道，又是兜底。P2P 是叠加的加速层，挂了不影响功能。
- **对称 NAT / 双重 NAT**：打洞失败是常态 → 回退兜底，用户无感（只是没拿到加速）。
- **直连中途断**（链路抖动/IP 变）：`p2p_ready` 复位 → 即时回退中继，可异步重新打洞。

## 5. 数据面选路（Phase 3 落地，已实现）

### 5.0 实现态（Phase 3，含对抗审查修复）
落地结构（实际）：`RemoteWs` 持 `p2p: Option<P2pEngine>`（引擎含 `data_tx`/`data_ready: Arc<AtomicBool>`）
+ `p2p_data_active: bool`（去重切换事件）。`send_frame` 选路 + 入站经 `P2pEvent::DataFrame` 汇回 `apply_relay`。

**关键修正（对抗审查 finding）——切换方向不对称**：
- **仅「被控端→控制端 输出方向」走 QUIC**；**「控制端→被控端 输入方向」恒走中继**。
- 原因：切换瞬间 QUIC 帧（1 跳）可越过在飞的中继帧（2 跳）造成乱序。**输出**乱序由切换时控制端补发
  `subscribe_tab` 触发的整屏快照重建自愈（文件块另有 seq 重组），故走 QUIC 安全；**输入**乱序会让被控端
  PTY 收到错序字节、污染 shell 输入且**无快照可自愈**——故输入永不走 QUIC。

`send_frame` 实际选路：
```rust
fn send_frame(&self, frame: &RemoteFrame) {
    let v = frame.to_value()?; // 失败仅 log
    // 仅被控端的非信令数据帧（输出方向）在直连就绪时走 QUIC；P2pSignal 信令必走中继。
    if self.is_controlled() && !matches!(frame, RemoteFrame::P2pSignal { .. }) {
        if let Some(p2p) = &self.p2p {
            if p2p.is_data_ready() && p2p.try_send_frame(v.clone()) { return; }
        }
    }
    self.send(RemoteC2S::Relay(v)); // 回退/默认/输入方向：中继
}
```
入站：P2P 读泵把帧经 `P2pEvent::DataFrame(Value)` 投回主线程，`apply_p2p_event` → `apply_relay(&v)`，与中继帧汇入同一状态机。

**`data_ready` 置位时机**：由 `run_data_plane` 在**本端数据面流真正建立后**置位（非据对端 Ready 信令），
避免流失败/未建时误判已直连（审查 #4）。切换/回退（`DataPlaneUp`/`Down`）经 `p2p_data_active` 去重，
控制端补发订阅重建镜像、被控端切/退输出方向。**`MAX_DATA_FRAME` 与中继 4 MiB 对齐**（审查 #2/#7，回退不击穿 WS）。

### 5.1 QUIC 流模型
- **当前（Phase 3）**：一条**双向 QUIC stream** 承载输出方向全部数据面帧（length-prefix：u32 大端 + JSON，
  复用 `to_value`/`from_value`，与中继字节一致）。Controller `open_bi` 后写 `P2pStreamHello` 首帧解除对端
  `accept_bi` 阻塞（quinn open_bi 惰性）。
- **Phase 4 优化**：文件块大流量改**独立单向 stream**，QUIC 多路复用避免「文件传输堵住输出」队头阻塞；
  输入方向若要走 QUIC，需先实现 drain-then-switch 屏障（中继发标记 + 对端 ack 后再切）或帧级 seq 重排。

## 6. 证书信任 / E2E 加密模型

- QUIC = TLS 1.3，双方各用 **rcgen 自签证书**（无 CA）。
- **信任锚 = 已鉴权的中继信令通道**：该 WS 会话已过 JWT + 配对鉴权，对端身份可信。双方经
  `Offer`/`Answer` 交换各自证书的 **SPKI 指纹（SHA-256）**。
- **直连握手时校验**：自定义 rustls `ServerCertVerifier`/`ClientCertVerifier`，只认「指纹 == 信令交换值」
  的对端证书 → **防 MITM**（攻击者即便在 NAT 路径上，无信令通道的指纹也无法冒充）。
- nonce 防重放：`Offer` 带随机 nonce，绑定本次打洞会话。
- **安全模型变化**：数据直连后端到端加密、**不经服务器** → 须重新安全审查（与路线图 §3.2 安全硬化
  一并做）。审查要点：指纹信任链、自签证书生命周期、降级攻击（强制 Fallback 是否削弱安全——答：中继
  路径本身经 TLS(wss)/JWT，回退不降级，仅丢失「不经服务器」属性）。

## 9. 风险与权衡

1. **tokio 引入**：编译时间/体积增加。缓解：隔离在 P2P 线程（§3），不侵蚀现有范式；tokio 已在 workspace lock。
2. **对称 NAT 打洞失败**：常态，非缺陷。缓解：中继兜底，用户无感（仅未获加速）。须 `log` 回退原因便于诊断。
3. **双栈维护成本**：数据面两条路（P2P/Relay），测试矩阵翻倍。缓解：选路抽象收敛在 `send_frame` 一处 +
   入站复用同一 `apply_relay`；**中继路径代码永不动**（P2P 是纯叠加）。
4. **安全模型变化**：E2E 直连绕过服务器。缓解：指纹信任锚（§6）+ 与 §3.2 安全审查合并评审。
5. **国内 STUN/UDP 可达性**：公共 STUN 可能不可达/慢。缓解：自建 server UDP 反射（§7）。
6. **静默回退**：打洞失败若无提示，用户以为在用直连。缓解：状态栏标识「直连/中继」+ 日志回退原因
   （路线图禁「无声降级」精神）。
