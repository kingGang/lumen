## 1. 能力与边界

- **单文件**：远程视图复制一个文件（快捷键 Ctrl+C / 右键复制）→ 资源管理器 / 桌面 / 任意应用
  Ctrl+V → 流式下载落地，进度条实时。
- **目录**：暂不进系统剪贴板（需递归枚举整棵子树平铺进 descriptor，阶段二）；复制目录时清掉系统侧
  残留虚拟文件、仅留 Lumen 内部剪贴板（Lumen 本机视图仍可下载整个目录）。

## 2. 架构（流式）

### 专用 STA OLE 线程（`virtual_files.rs`）
- `ClipboardService::start(proxy, wake_pending, fetch_tx)`：起专用线程 `OleInitialize`（STA）+ 消息泵 +
  `OleSetClipboard`，**不污染** winit 主循环。UI 主线程经句柄发命令（`set_remote_file` / `clear`），
  `PostThreadMessageW(WM_APP)` 唤醒消息泵 drain 命令。`Drop` 时 `Stop` + `PostQuitMessage` + `join`。
- `RemoteDataObject`（`#[implement(IDataObject)]`）暴露：`FileGroupDescriptorW`（文件名 + `FD_PROGRESSUI`，
  **不设 FD_FILESIZE**）/ `FileContents`（`TYMED_ISTREAM`）/ `Preferred DropEffect`（COPY）。
- `RemoteFileStream`（`#[implement(IStream)]`）：自实现流式只读流，`Read` 阻塞消费 [`StreamMsg`]。

### 数据流（资源管理器粘贴远程文件）
```
资源管理器 Ctrl+V
  → OLE 线程 IDataObject::GetData(FileContents)
    → 立即创建 RemoteFileStream + 发 ClipFetchCmd{path,data_tx} + nudge(PtyWake) → 立即返回流（不阻塞！）
  → 资源管理器拿到流即开始 copy（弹进度对话框）→ 调 IStream::Read（编组回 OLE 线程，阻塞等数据）
  → UI 主线程 user_event → drain clip_fetch_rx → RemoteWs::start_clip_fetch → 发 FetchReq（FetchKind::Clipboard）
  → 被控端 fetch_src_worker 流式读发 FileChunk（零改）
  → UI 主线程 fetch_chunk 把分块 data_tx.send(StreamMsg::Chunk) 喂流 + 回 FileChunkAck（背压窗口 4）
  → OLE 线程 RemoteFileStream::Read recv 到块 → 返回给资源管理器 → 进度条推进
  → FileEnd → fetch_end 发 StreamMsg::Done → Read 读到即 EOF → 资源管理器完成落地
```
- **不阻塞**：`GetData` 立即返回流，资源管理器立即显示进度 → 进度随真实下载推进，**无黑等**。
- **无死锁**：`Read` 在 OLE 线程阻塞等数据，由 UI 主线程喂；UI 主线程从不反向阻塞 OLE 线程。
- **取消**：用户取消 → 资源管理器 Release 流 → `data_tx` 对端 drop → 下个 `fetch_chunk` send 失败 →
  `fetch_abort` 发 `FetchCancel` 回收被控端源 worker。
- **失败**：对端 `FileErr` / 乱序 / 停滞 → `fetch_abort` 发 `StreamMsg::Failed` → `Read` 返回 `E_FAIL` →
  资源管理器粘贴报错（**不落不完整文件**）。

## 4. 已知限制（MVP 边界）

1. **仅单文件**：目录不进系统剪贴板；Lumen 内仍可下载目录。
2. **进度条无精确总量**：descriptor 未设 `FD_FILESIZE`、`IStream::Stat` 返回 `cbSize=0`（流式开始时不
   知道总大小）。资源管理器会显示「已复制 N 字节」活动但百分比可能不精确。阶段二透传 size 即可精确。
3. **背压为「立即 ACK」**：`fetch_chunk` 收到块即喂 channel + 回 ACK，被控端全速发。资源管理器读得快时
   channel 几乎不积压；但**用户暂停**或**目标盘很慢**时，分块会在内存 channel 积压（上界=文件大小）。
   阶段二改「消费驱动 ACK」（Read 读走才 ACK）把内存上界压到一个窗口。
4. **进程必须存活**：延迟渲染本质——关掉 Lumen 后系统剪贴板里的虚拟文件失效，粘贴会失败。
5. **会话断开**：断开后粘贴因 `start_clip_fetch` 的 `is_controlling` 守卫喂 `Failed`（资源管理器报错），
   不崩；登出时主动 `clear` 系统剪贴板。
