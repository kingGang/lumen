## ✅ 已修复（待发版）：窗格最大化还原后「其它窗格」内容串扰 bug

**现象**（海风哥实测截图证实）：分屏多窗格时，**点窗格标题栏「还原」按钮**（从最大化态还原），其余被遮挡窗格内容错乱——**每行只剩第一个字符，其余内容永久丢失**。

**根因**（全在 `crates/lumen-app/src/main.rs` 的 RedrawRequested 流程，是 egui 帧内状态变更与 resize 时序错位的竞态）：
1. `4676` 构造 `ShellInput.maximized = tab.maximized`（run_ui 起点快照）。
2. run_ui 内 `shell::show`（`shell/mod.rs:720-742`）按 `input.maximized` 算 `pane_rects`：最大化态隐藏窗格 = `egui::Rect::NOTHING`（退化矩形，宽高 -∞）。
3. run_ui 内点「还原」按钮 → `shell_out.pane_maximize = Some(pi)`。
4. `5385` run_ui 之后处理它 → `toggle_maximize_pane` → `state.tabs[].maximized = None`。
5. `6006-6188` resize 循环：原 `6035` 读**实时** maximized（已 None）判跳过隐藏窗格 → 判定失效 → 拿 NOTHING 矩形 → `(-∞).max(1.0)` 夹成 `tw=th=1` → grid resize 成 **1 列** → `grid.rs` `row.resize(1)` 截断每行到首字符、scrollback 右侧永久丢失。
- **为何按钮中招、快捷键不中**：`Ctrl+Shift+Enter`（`3807`）走事件层、下一帧才 run_ui，`input` 与判定同源；按钮走 egui 内、同帧改状态致失同步。

**修复**（`main.rs:6029-6055`，8 行）：resize 循环不再读"可能帧内被改掉的 maximized 状态"，改以 **`pane_rects` 几何本身**为准——退化矩形（NOTHING/非有限/宽高 <1pt）一律 `continue` 跳过，留待下一帧用正确矩形 resize：
```rust
let (rw, rh) = (r.width(), r.height());
if !(rw.is_finite() && rh.is_finite() && rw >= 1.0 && rh >= 1.0) { continue; }
```
**为何选"判矩形"而非"判状态快照"**：对抗审查发现 `layout_reset`（工具栏「▦」一键复位按钮，`reset_pane_layout`）是**同根因第二入口**——它在最大化态也会帧内清 maximized；判矩形方案**一并覆盖**了它，若按状态快照修则会漏。

**防御加固**（同会话已加注释，防未来回归）：① `shell/mod.rs` 产 NOTHING 处加「契约·勿改」（隐藏窗格必须产退化矩形，勿改成 0/正尺寸真矩形绕过 guard）；② `toggle_maximize_pane`/`reset_pane_layout` 两个还原分支加「补帧三件套·勿拆」（清 maximized + 全窗格超龄 `term_frame_due_since` + `request_redraw` 须一致，新增「帧内清 maximized」动作必须复制）。

**验证**：真机按钮路径自动复现脚本（`ptylog/maximize_verify.ps1`，还原走点 `⧉` 按钮坐标）还原后被遮挡长行窗格内容完整无损 ✓；海风哥手动还原确认 ✓；`clippy` 0 warning ✓；全测试 467 项通过 ✓；对抗审查 workflow（18 agents、14 发现/8 确认全为 low「核实通过」、零需补修）✓。
> 复现/验证脚本与取证：`ptylog/maximize_repro.ps1`（快捷键路径·不触发）、`maximize_verify.ps1`（按钮路径·触发）、`review_wf.mjs`（对抗审查 workflow）。窗格还原按钮 `⧉` 在窗格标题栏右端（最大化态约窗口右缘内 33px、顶下 46px；那是窗格标题栏行，app 顶栏窗控在更上面 y≈16）。

## ✅ 已完成：终端区可拖动滚动条（2026-06-14，待发版）

**需求**：命令行（终端工作区）加**可拖动滚动条**。已实现并真机验证通过，**尚未 bump 版本号/发版**（等海风哥拍板）。

**实现**（全部在 `crates/lumen-app/src/main.rs`，仿「回到底部按钮」overlay 套路）：
- `struct ScrollbarGeom { sid, track, thumb, scrollback }`：逐窗格几何。
- `AppState.scrollbar_drag: Option<(SessionId, f32)>`：拖动态（会话 id + 抓取锚点，跟手不跳）。
- `fn scrollbar_overlays(&self)`：只对 scrollback 非空、内容区够高的可见窗格出条；几何取上一帧 `pane_rects_px`（物理像素→逻辑点），焦点窗格扣 footer，轨道贴右缘内侧宽 10pt。滑块模型：`total=scrollback+rows`，滑块高=可视占比 `rows/total`（最小 28pt），位置按 `display_offset`（0=贴底、`scrollback`=贴顶）。`thumb_top = ty + (1 - offset/sb)*movable`。
- run_ui 闭包内逐条 `egui::Area`(Foreground)+`Sense::click_and_drag()`：拖滑块跟手、点轨道空白跳转、`interact_pointer_pos` 绝对反算 offset（无漂移）；idle alpha 0.40、hover/拖动 0.85+淡轨道底、胶囊圆角、配色 `fg_dim`。
- 闭包后按「目标 offset − 当前 offset」走现成 `grid.scroll_display(delta)`（**没动 lumen-term**）。

**对抗式审查修复的 3 类问题**（workflow 12 agents 交叉验证）：
1. **scrollbar_drag 泄漏兜底**：唯一清零点 `drag_stopped()` 在循环内，被拖窗格拖动中从 targets 掉出（切 tab/关窗格/清屏/变矮）或失焦/Alt+Tab 时永不触发→滚动条恒高亮。已在闭包后加 `!pointer.any_down()` 无条件兜底（复用 divider 惯例）。
2. **滚轮死区**：轨道是 Foreground 层→`pane_under_mouse` 返回 None→窗格右缘整列滚不动。已加 `mouse_on_scrollbar_track()`（复用 `scrollbar_overlays` 几何），MouseWheel 门补判放行。
3. **点击滑块跳变**：click 分支没判 `thumb.contains(p)`→点滑块本体窜走。已镜像 drag-start 的判断。

**真机验证**（脚本在 `ptylog/`：`verify_scrollbar.ps1` 显示+滚轮、`drag_scrollbar.ps1` 拖动、`analyze_scrollbar.ps1` 像素定量）：宽 10px 贴右缘 ✓；idle RGB(65)=fg_dim×0.40、hover RGB(134)=×0.85 ✓；滑块高占比 14%(=可视占比) ✓；滚轮滚到中部滑块在中段、拖到顶贴顶、拖到底贴底、跟手到位 ✓。
> 真机验证 harness 坑：截图进程须**先**调 `SetProcessDpiAwarenessContext(-4)` 再加载 Forms/Drawing（否则 DPI 已初始化、设置失败→GetWindowRect 物理 与 CopyFromScreen 虚拟 坐标错位）；窗口最大化用 GetWindowRect 让坐标自洽；SendKeys 注入命令前须**先点终端区**拿键盘焦点（terminal_focused），否则键被 egui 吃掉；像素扫描定位滑块要按「灰色调 R≈G≈B」连续段，否则被终端文字污染。
