# M5.3 part3d 远程「多会话 × 多窗格」镜像 设计稿（2026-06-22）

> 经「理解 + 设计 workflow（3 维度设计 + 1 对抗审查）」产出。本稿是动手前的方案 + 分阶段 + 待海风哥
> 拍板的决策点。server 数据面**零改**（盲转 `serde_json::Value`）。

## 0. 需求（海风哥已确认）

控制端要做到：(a) 看到被控端**所有会话(tab)**列表 + 状态（名/cwd/标题/忙/未读），随创建/删除实时更新；
(b) 每会话内的**多窗格(pane 分屏)**布局 + 各窗格内容；(c) 点击切换看任一会话 → 被控端按**订阅**推该
会话内容（被控端焦点不动）；(d) 控制端能**新建/关闭**被控端会话；(e) 控制端选中与被控端焦点**互不同步**。

现状：控制端只镜像被控端**单个焦点窗格**（`mirror: Option<Terminal>`，`Output(Vec<u8>)` 无 id）。

## 1. 对抗审查的两大要害（三份设计都低估）

- **D1（ID 模型）**：代码里 `TabId` 与 `SessionId` 是**两个独立自增空间**，一个 `Tab` 含 `Vec<Session>`
  （每个 pane = 一个 `Session`，有自己稳定的 `SessionId`）。三份设计混淆 tab/pane id。**关窗格会让
  `Vec<Session>` 下标重排** → 若用下标做路由/缓存键会静默错位内容。**裁决：用 `(TabId, SessionId)` 双 id
  路由，`pane_index` 仅作渲染顺序、绝不做路由/缓存键**（与现有 `pane_rects_px: Vec<(SessionId,...)>` 一致）。
- **D2（控制端渲染真相）**：控制端镜像当前渲染为**一张满屏离屏纹理 + `egui::Image`**（`MIRROR_OFFSCREEN_ID`），
  **不走**本地多窗格 `pane_rects` 路径（那条吃真实 `Session`+PTY，镜像侧没有）。**多窗格镜像必须新建渲染
  路径**（复刻 `pane_rects` 几何 + 分隔条 + 标题栏 + 最大化），是真重写、不是「复用本地路径」。难点不在
  协议，在这里。

其余风险：D3 订阅切换快照保序（快照必先于该会话任何输出）；D4 状态去重（layout 浮点 + spinner 标题每帧
抖动会刷爆链路，去重键须排除高频抖动字段）；D5 旧 `Output` 兼容是伪命题（两端同 app 同版本，一次性切换 +
版本门）；D6 本地/远程 tab id 命名空间分流、重连用排序而非 HashMap 迭代序；D7 后台订阅窗格 PTY 无 ACK
背压需节流。

## 2. 分阶段实现计划（每阶段可独立交付 + 可测）

- **Phase 0 协议地基**（先拍 K1/K2，纯协议 + 往返/满载测试，无 UI）：加会话/窗格状态帧
  `TabListSnapshot/TabCreated/TabClosed/TabUpdated`、订阅 `SubscribeSession`/`SubscriptionStarted`、
  内容 `OutputWithId/ResizeWithId`、增删 `NewTab/CloseTab`(+结果)。
- **Phase 1 MVP**（满足 (a)(c)(e) 最小闭环，**零渲染重构**）：被控端推会话列表 + 状态（埋全 dirty 钩子 +
  去重）；控制端会话栏切到远程列表、点击发 `SubscribeSession`；被控端回快照但 **MVP 只取焦点窗格那 1 个**
  喂进**现有单 `mirror` 纹理**（D2 渲染不动）；**明确只读**（不发输入，规避 D6 焦点路由）。两机可测：被控端
  开 3 会话，控制端看列表+点切换看对应会话焦点窗格，被控端在别的会话干活互不扰。
- **Phase 2 增删会话**（需求 d）：`NewTab/CloseTab` + 结果；超 `REMOTE_MAX_SESSIONS` 回失败；切的会话被关
  → 控制端按排序回退。
- **Phase 3 多窗格渲染**（需求 b，**最大重构、单独成阶段**）：控制端 `HashMap<SessionId, Terminal>`
  per-pane；**新建镜像多窗格渲染路径**（D2）；`SubscriptionStarted` 带全部窗格快照 + 保序锁（D3）；被控端
  双路 tee（焦点 + 订阅会话全部窗格）+ 输出节流（D7）。
- **Phase 4 历史回看 per-pane + 控制端输入路由**（需求 e 的「写」）：`HistoryReqForPane`；`Input` → 带
  `(tab_id, session_id)` 写指定窗格而非焦点。

Phase 1 即满足核心三需求且零渲染重构；Phase 3 才碰渲染大重构，风险隔离。

## 3. 待海风哥拍板的决策点（K1–K6，附推荐）

| 键 | 决策 | 推荐 |
|---|---|---|
| **K1** | ID 路由模型 | **`(TabId, SessionId)` 双 id；`pane_index` 仅渲染序、不做路由/缓存键** |
| **K2** | 旧 `Output` 帧兼容 | **一次性切到带 id 帧 + `Welcome.min_supported_version` 版本门**（不双发灰度） |
| **K3** | 会话栏形态 | **远程视图整组替换为远程会话列表**（最简、无 id 串扰） |
| **K4** | 多窗格渲染路径 | **MVP 先只渲染订阅会话的焦点窗格（复用现有单纹理）；多窗格独立 Phase 3** |
| **K5** | 订阅并发 + 节流 | **同时只订阅 1 个会话 + 镜像输出加帧合并/节流** |
| **K6** | 状态去重粒度 | **去重键排除 layout 浮点 + 归一化 spinner 标题；布局变化走低频帧** |
