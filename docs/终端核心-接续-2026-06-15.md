## 1. req6：窗口缩小再放大，文本宽度不复原（主要在未显示的 scrollback 历史区）

### 现象（海风哥口径）
把窗口缩小再放大，文本宽度没复原，**主要在「没被显示的部分」（scrollback 历史区）**——回看历史时是窄的。

### 已证伪的假设（本会话用 `[R6-RESIZE]` 实测日志 + 读码排除，别再往这查）
- main.rs 的 resize 循环 + grid 尺寸 resize **是对的**。实测放大帧：`tw/th` 正确变大、目标 grid `41x65`、`divider_held=false`、`term.resize`/`pty.resize` 已提交。
- 因此以下全部**证伪**：x0/x1 各自 round 抖动、`grid_size_for` 有 cache 跳过、footer height debounce、`divider_resize_held` 门控误触（H1）、`window_just_resized` 被非 resize 帧吞（H2）、ppp 陈旧（H3）。

### 真根因（`crates/lumen-term/src/grid.rs` `Grid::resize`，搜 `pub fn resize`）
1. 改列宽时只对**屏幕行**做 reflow：`for row in &mut self.screen { row.resize(cols); }`——**完全不碰 `self.scrollback` 历史行**。
2. `Row::resize`（搜 `fn resize` in `impl Row`）= `cells.resize(cols, default)`：**缩列直接截断丢右侧内容**。
3. `Row` 只有 `cells: Vec<Cell>`（搜 `struct Row`），**无折行元数据**——无法判断哪些行是同一逻辑行的折行续行，故**真正的 reflow（解折行/重排）必须先给 Row/Grid 加 wrap 标记**。
- 净效果：缩小期间产出 / 被推入 scrollback 的行是窄宽（右侧还被截断丢失），放大后**永不重排** → 回看历史是窄的。WT 会 reflow 历史并锚底，Lumen 不会。

### B3-6 历史包袱（**开工前必读** `grid.rs` 的 `Grid::resize` rustdoc，搜 `B3-6 撤销 B3-3`）
B3-3 曾做「放大时从 scrollback 拉回历史行」，**B3-6 撤销了**，原因：
1. **历史逐步蒸发**：ConPTY resize 后发整屏重绘流（`ESC[H` 归位 + `ESC[K` 逐行清空），把拉回的历史行清成空行；随后缩行时 B2「收割光标下空行不入历史」把这些**当真空行丢弃** → 反复缩放历史越丢越多。
2. **多余提示符行**：resize 后注入 `\r` 堆出多行 `PS F:\...>`——**此因已拆除**（B3-7，搜 `B3-7`），所以重开拉回主要剩**风险①蒸发**。

### 修法方向（需谨慎设计 + 真机回归测「反复缩放历史不丢」）
- 正途：实现 scrollback **reflow**（加 wrap 元数据 + 重排算法，参考 alacritty/wezterm）。工程量大、动最敏感核心。
- 或：重新引入 B3-3 式锚底/拉回，但加**防蒸发**保护（如标记拉回行、让 B2 收割不丢被清空的历史行）。
- **勿动** `crates/lumen-app/src/shell/mod.rs` 窗格循环里「隐藏窗格必须产 `egui::Rect::NOTHING`」的契约（搜 `NOTHING`，约 750-765）——那是「最大化还原每行剩首字符」串扰 bug 的根因修复。

## 2. R7：多 agent 视图折叠返回后下方一大块留白（需 2 窗格才复现，切目录树/任意 resize 才复原）

### 现象
Claude Code 进多 agent 列表（满高 TUI），返回主显示时下方留一大块空白；**切一下目录树面板（=一次 resize）就复原**。**1 个窗格不复现，要 2 个窗格才复现**。

### 真状态（本会话 `[R7-RENDER]` 实测日志）
- bug 态（折叠后持续）：`cursor_row=1, trailing_blank=25, disp_off=0, sb_len=276` —— 屏幕底部**真有 25 行空白**、内容+光标锚在**顶部**、**`disp_off=0`（排除 display_offset/视口错位修法）**、上方有 276 行 scrollback。
- 一次 resize 后：`cursor_row=25, trailing_blank=1`（内容填满）——是 **ConPTY 重发流**填的，不是 grid 结构变化（resize 改的是列、行没变）。

### 本质 = 与 req6 同一块核心
折叠后 Lumen 把短内容锚**顶部**留白；WT 锚**底部**（历史填上方、内容贴底）。「切 resize 修好」是 ConPTY 重发触发的副作用。**建议与 req6 一并设计**（同属 scrollback 锚定/填充）。

## 3. composer：切 tab 后首个中文字缩到最左、删不掉（仅 Win10、每次必现）

### 现象
**仅 Windows 10（Win11 不复现）**：切到别的 tab 再切回，点 composer 用**中文输入法**打字，**第一个字**缩到最左、删不掉。每次必现。

### 真根因（H1，~70%，IME 调查工作流 + 核对本机 winit 0.30.13 / egui-winit 0.34.3 源码）
切 tab 回来点 composer 时，**首个 `Ime::Preedit` 在 `terminal_focused` 翻 true 之前到达** → `bypass_egui=false`（搜 `let bypass_egui`）→ 事件喂给 egui → egui 把首字组合串画在它默认控件位（≈最左）；而 Lumen 的 `Ime::Preedit` 处理在 `!terminal_focused` 时**直接 return 丢弃**（搜 `WindowEvent::Ime(Ime::Preedit`）→ 字**从未进编辑器文档** → 退格删不掉。Win10 经典 IMM 与 Win11 TSF 的 `WM_IME_STARTCOMPOSITION/COMPOSITION` 投递时序差异 → **仅 Win10**。

### 已纠正的错误假设
tracer 曾推「Win10 OS 自绘组合串」——**不成立**：winit 0.30.13 `event_loop.rs` 对 `WM_IME_COMPOSITION` **无条件** `ProcResult::Value(0)` 不调 DefWindowProc，且 `WM_IME_SETCONTEXT` 抹掉 `ISC_SHOWUICOMPOSITIONWINDOW`，**OS 不会自绘**。所以那个字只能是 egui 画的。

### 坐实方法（本开发机是 Win11、复现不了，**必须 Win10 真机日志**）
重加下方 IME 诊断日志 → Win10 复现一次 → 看「切 tab 回打首个中文字」时**首个 `Ime::Preedit` 的 `bypass_egui` 是否 false / 是否没进 Lumen 的 Preedit 处理**。是 → 坐实 H1。

### 修法方向（需 Win10 + Win11 **双机**验证）
- `activate()` 切 tab 时复位 `preedit` / IME 状态；
- 或放宽 `Ime::Preedit` 的 `!terminal_focused` 丢弃：composer 可用时首个 Preedit 即便在焦点翻转窗口期也归 Lumen、不漏给 egui。
- **注意**：别让 egui 设置页/登录框里打字与终端双投。
